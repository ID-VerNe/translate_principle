# 🚀 AI 交互式深度翻译工作流手册 (Web UI 版)

本工作流旨在通过**分步引导**和**人机协作**，将英文技术文档/论文/博客转化为高质量、符合中文阅读习惯的科普文章。

### 核心优势
*   **上下文记忆**：利用对话窗口的记忆能力，无需重复粘贴上一步结果。
*   **过程可控**：每一步输出后，都可以进行人工干预（如修正术语），防止错误累积。
*   **质量保证**：包含专门的“找茬”与“注释”环节，确保译文准确且有深度。

---

## 🛠️ 准备工作
1.  打开 AI 网页端（Gemini 3 Pro）。
2.  **新建一个对话窗口**（每个新文档建议新建对话）。
3.  按顺序执行以下步骤。

---

## 0. 初始化 (System Prompt)
**操作**：复制下方内容发送给 AI，为它设定“员工手册”。

```markdown
# Role (角色设定)
你是一名精通简体中文的专业技术译者，特别擅长将专业的英文学术论文或技术文档转换为通俗易懂的中文科普文章。

# Global Rules (全局规范)
在接下来的对话中，我们将对一段英文文本进行翻译。无论进行到哪一步，你都必须严格遵守以下规范：
1. **人名处理**：技术书籍中的人名通常不翻译，除非是众所周知的（如乔布斯）。
2. **书名处理**：有中文版的用中文版书名；无中文版的直接保留英文书名。
3. **术语格式**：英文术语首次出现时，若有必要，应使用“HTML（Hypertext Markup Language，超文本标识语言）”的格式，之后可使用简写。
4. **代码处理**：代码块（Code Block）完全不翻译。代码内的注释需翻译（中英对照）。需标注“原书Pxx页代码xx”。
5. **标点符号**：译文必须遵循中文标点符号的使用习惯，严禁照搬英文标点。
6. **图表**：图题、表题需翻译。
```

---

## 1. 第一步：术语提取 (Term Extraction)
**操作**：复制下方指令，并将 `{{原文}}` 替换为你要翻译的英文内容。

```markdown
<任务> 识别用户输入文本中的技术术语。请严格按照示例中的格式，展示翻译前后的技术术语对应关系。

<示例>
| 英文 | 中文 |
| --- | --- |
| Prompt Engineering | 提示词工程 |
| Zero Shot | 零样本 |
| Context | 上下文 |

<输入文本> 
{{此处粘贴英文原文}}
```

> **⚡️ 人工干预点（重要）：**
> AI 输出表格后，请快速扫一眼。如果发现关键术语翻译不准确（例如将 LLM 中的 "Temperature" 译为 "温度" 而非 "随机性"），请直接回复 AI：“**把 Temperature 改成‘随机性’，其他的保持不变。**” 或直接修改AI的回复内容（大部分WebUI都允许用户手动修改AI回复），确认无误后再进行下一步。

---

## 2. 第二步：直译 (Straight Translation)
**操作**：直接发送下方指令（无需粘贴原文，AI 会记得）。

```markdown
<任务> 接下来进行第二步：直译。

请参考**上文生成的术语表格**，以及**最开始的输入文本**，进行逐句直译。

<限制> 
1. 维持原有的段落格式，不省略任何信息。
2. 优先保证准确性，暂不需要过度润色。
3. 遇到上文表格中的术语，请严格按照表格对应的中文进行翻译。

请直接输出直译结果：
```

---

## 3. 第三步：找茬与注释规划 (Issue Spotting)
**操作**：发送下方指令。这一步是提升质量的关键。

```markdown
<任务> 接下来进行第三步：审校找茬。

请作为审校专家，检查**刚刚生成的直译文本**，结合**英文原文**，指出其中具体存在的问题。

<检查标准>
1. **表达习惯**：指出不符合中文表达习惯或句子结构笨拙的位置。
2. **规范检查**：
   - 人名是否保留英文（除众所周知外）？
   - 书名是否有中文版？无中文版需保留英文。
   - 首次出现的英文术语是否使用了“HTML（Hypertext Markup Language，超文本标识语言）”格式？
   - 标点符号是否已转换为中文标点？
3. **代码与图表**：检查代码注释是否翻译，代码本体是否保留。
4. **注释规划**：请找出文中包含了“文化梗、特定的行业黑话、只有当地人懂的地理位置或历史背景”的地方。如果直接翻译读者可能看不懂，请标记需要添加“译者注”。

<输出要求>
1. 列出具体的问题清单，不需要修改原文，只需列出问题。
2. 列出**拟添加注释的清单**（格式：原文词汇 -> 拟解释内容）。
```

---

## 4. 第四步：最终意译 (Final Polish)
**操作**：根据文章类型，先选择A发送，如果有需要，可以在A回复完成后，发送B进行进一步风格化润色。

### A：标准科普风格（适用于技术文档/论文）
```markdown
<任务> 接下来进行第四步：最终意译。

请基于**上一步指出的问题清单**，对**之前的直译文本**进行修正和润色。

<要求>
1. **风格调整**：将语风转化为“通俗易懂的科普文章”。
2. **问题修正**：必须逐一解决上文指出的所有问题。
3. **格式保留**：保持原有的段落和结构不变。
4. **输出限制**：**只输出最终的翻译文本**，不需要写任何多余的话语（如“好的，这是最终结果”）。
5. **插入注释**：在正文中需要解释的地方插入 `[注x]` 标记（x为序号）。
6. **文末列表**：在文章最后，生成一个 Markdown 分割线，然后列出所有注释内容。
   格式示例：
   ---
   **译者注**
   [注1] **Bilibili**：中国知名的视频弹幕网站。
   [注2] **404**：指网页未找到错误。

<输出限制>
只输出正文和文末的注释列表，不要其他废话。
```

### B：特调风格版（适用于博客/观点/特殊作者）
*需要预先定义作者风格，以下以 Clarkson 为例*

```markdown
<任务> 最终意译（克拉克森特调版 + 客观注释）。

<核心指令>
原文作者是 **Jeremy Clarkson**，他的风格是：**尖酸刻薄、极度夸张、英式幽默、充满比喻**。

<角色分离设定>
1. **正文部分**：请完全沉浸在 **Jeremy Clarkson** 的角色中，用第一人称（“我”）进行翻译，风格要辛辣、口语化。
2. **注释部分**：请切换回 **客观译者** 的身份（第三人称）。**严禁**在注释中使用“我”或模仿克拉克森的语气。注释必须是冷静、客观的背景知识补充。

<执行要求>
1. **用词**：正文使用更有冲击力的词汇（例如把 "uncomfortable" 译为 "把你的脊椎骨都要颠散架了"）。
2. **注释内容**：遇到英国文化梗（如 Milibandification, giddy aunt），在正文标 `[注x]`，并在文末进行解释。
   - *错误示范*：[注1] 这是我发明的一个词，用来嘲笑那些人……
   - *正确示范*：[注1] **米利班德化**：作者自创词。指英国工党前党魁埃德·米利班德（Ed Miliband），因其激进的环保政策常被克拉克森嘲讽。此处指为了环保而牺牲性能。
3. **格式**：保持 Markdown 格式，文末使用分割线区分正文与注释。

请输出最终结果：
```
